# blind-spot-scan

在规格写完之后、动手写代码之前，主动做一次认知盲区扫描，把「你不知道自己不知道」和「你见过才认得出但没写下来」的东西提前摆到桌面上。

## 概述

实现质量的最大瓶颈往往不是模型能力，而是你对问题空间的认知盲区。规格文档覆盖了「已知的已知」和部分「已知的未知」，但有三类东西天然容易被遗漏：

- **未知的已知**：你见过才认得出的标准和要求（UI 手感、交互节奏、边界习惯），但不会主动写进规格。
- **未知的未知**：你完全没考虑过的方向（架构隐含约束、跨端交互、极端场景）。
- **规格与现实的落差**：规格写了某个行为，但现有代码库的架构决定了这个行为需要付出你未预估的代价。

这个技能的作用，就是在代价最低的原型阶段，把这些东西找出来。

## 何时触发

- 用户刚写完一份规格文档（`specs/NN-xxx/PRODUCT.md` 和/或 `TECH.md`），准备进入实现阶段。
- 用户主动提到「盲区」「有没有漏掉什么」「我没想到的」。
- 进入一个新的功能模块实现前，自动建议做一次扫描。

**不需要触发的情况**：简单的 bug 修复、单文件修改、已实现过的类似功能。

## 工作流程

### 1. 确定扫描目标

读取 `specs/` 目录，找到最新的规格目录（按编号排序取最大）。如果用户指定了具体目录，使用用户指定的。

读取该目录下的 `PRODUCT.md` 和 `TECH.md`，理解当前规格覆盖的范围。

使用 `codegraph_explore` 扫描项目代码库中与该功能相关的现有代码结构，理解实现这个规格需要触碰的现有模块。

### 2. 四象限分析

基于规格内容 + 代码库现状，输出以下四个部分的分析：

#### 已知的已知（规格已覆盖）

列出规格中已明确写明的核心行为和约束。这不是重点——重点是确认你对规格的理解和规格本身一致。简要列点即可。

#### 已知的未知（规格已标记但未解决）

列出规格中显式标为开放问题（**开放问题：**）、TODO、或明显留白的地方。这些是你已经意识到「这里还没想清楚」的部分。

#### 未知的已知（推测你可能有的隐含期望）

根据你对这个项目的已有决策风格（参考 `docs/01-需求/决策记录.md` 中的模式）和代码库现状，推测你可能有的、但没写进规格的隐性要求。这些是你「见了才会认出来」的标准。

思考方向：
- 同类型功能在项目中已经有的交互模式（比如错误处理是抛异常还是返回错误码）
- 你的决策记录中体现的偏好（比如偏向简单方案、偏向零依赖）
- 代码库中相邻模块已有的约定（比如日志格式、配置命名、错误码范围）
- 用户体验层面的隐含期望（比如「断线了用户应该知道」→ 是否需要 toast？）

#### 未知的未知（你大概率还没考虑到）

根据功能复杂度和项目上下文，列出你可能完全没考虑到的方向。思考方向：
- 该功能涉及哪些外部系统/依赖，它们的故障模式你考虑了吗？
- 多租户 / 并发 / 网络异常等跨院场景下的边界？
- 与项目其他 specs 中已定义的功能的交集和冲突？
- 与现有代码架构的兼容性问题（比如新功能需要的连接池会不会和 WebSocket 长连接冲突）？
- 一期不实现但会影响接口设计的二期预留？

### 3. 原型建议

对每个「未知的已知」和「未知的未知」，判断是否可以用一个最小原型快速验证。优先推荐能用单个 HTML artifact 回答的问题。

原型建议格式：
- **问题**：你还不确定的具体点是什么
- **原型方式**：一个 HTML mockup / 一段假数据的接口调用 / 一个配置项
- **为什么值得做**：在原型阶段发现这个问题的代价 vs 实现后发现

如果某个问题无法用低成本原型验证，标注为「需直接讨论」。

### 4. 最该先问的三个问题

从以上分析中，选出**改了会改变整体架构**的三个问题，按影响程度排序。每个问题附带：
- 为什么重要
- 不回答会有什么默认假设被做出

### 5. 输出格式

用以下 Markdown 结构输出：

```markdown
# 盲区扫描：[功能名称]

> 扫描时间：{日期}
> 规格来源：specs/NN-xxx/

## 已知的已知
- ...

## 已知的未知
- ...

## 推测的未知的已知
- ...

## 未知的未知
- ...

## 原型建议
| 问题 | 原型方式 | 为什么值得做 |
|------|---------|------------|
| ...  | ...     | ...          |

## 最该先问的三个问题
1. ...
2. ...
3. ...
```

如果分析后有原型建议，**额外提供一个 HTML artifact** 来可视化呈现分析结果，方便用户快速浏览和决策。

## 与现有工作流的衔接

这个技能不是孤立使用的。它插在现有流程中：

```
写规格（write-product-spec / write-tech-spec）
    ↓
盲区扫描（本技能）← 你在这里
    ↓
原型验证（如需要）
    ↓
实施计划
    ↓
实现（implement-specs）
```

触发本技能后，根据扫描结果：
- 如果「已知的未知」和「未知的已知」都很少 → 直接进入实施计划
- 如果有原型建议 → 先做原型，用户确认后再进入实施计划
- 如果发现了影响架构的未知数 → 先讨论，必要时回写规格文档

## 相关技能

- `spec-driven-implementation` — 本技能的上游（写规格）和下游（实现）
- `write-product-spec` — 写 PRODUCT.md
- `write-tech-spec` — 写 TECH.md
- `implement-specs` — 根据已批准的规格实施
