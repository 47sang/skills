# write-tech-spec

为重要功能编写 `TECH.md` 规格说明。

## 概述

技术规格应将产品意图转化为适合现有代码库的实现计划，记录架构选择，并使代理更容易执行、评审更容易评估。

将规格写入 `specs/01-门诊混发/TECH.md`，其中 `01-门诊混发` 可以是以下之一：
- Linear 工单号（例如 `specs/01-门诊混发/TECH.md`）
- GitHub issue id，带有 `gh-` 前缀（例如 `specs/gh-4567/TECH.md`）
- 短横线命名的功能名称（例如 `specs/01-门诊混发/TECH.md`）

当存在同级的 `PRODUCT.md` 时，匹配其使用的编号-描述。`specs/` 应仅包含以编号-描述命名的目录作为直接子级。

工单/issue 引用是可选的。如果用户有 Linear 工单或 GitHub issue，使用其 id。如果没有，请他们提供用于目录的功能名称。仅在用户明确要求时才创建新的 Linear 工单或 GitHub issue；在这种情况下使用 Linear MCP 工具或 `gh` CLI（如果团队、标签或仓库不明确，使用 `ask_user_question`）。

## 何时使用

当实现跨越多个模块、有意义的架构权衡，或评审者在看到代码之前或同时查看计划会受益时，使用此技能。对于纯 UI 更改或简单的修复，通常不需要技术规格。

最好先有 `PRODUCT.md`，以便技术计划锚定于商定的行为。如果实现仍然太不确定，先构建端到端原型，然后从中学到的东西编写技术规格。

## 编写前的研究

在起草之前，阅读产品规格（如果有的话），检查相关代码，并确定主要文件、类型、数据流和所有权边界。当代码可以直接检查时，不要猜测当前架构。

## 结构

必选部分：

1. **上下文** — 正在构建什么、被更改区域的当前系统如何工作，以及带有行引用的最相关文件。将"问题"、"当前状态"和"相关代码"合并为一个有依据的部分。示例引用：
   - `app/src/workspace/mod.rs:42` — 用户流程的入口点
   - `app/src/workspace/workspace.rs (120-220)` — 可能更改的状态和事件处理
   引用 `PRODUCT.md` 以获取用户可见行为，而不是重新陈述它。

2. **建议的更改** — 实现计划：哪些模块会更改、引入的新类型/API/状态、数据流、所有权边界，以及设计如何遵循现有模式。当存在多个合理路径时，指出权衡。

3. **测试和验证** — 实现将如何根据产品行为进行验证。负责证明功能正常工作的一切：单元测试、集成测试、手动步骤、截图、视频以及任何其他验证。直接引用 `PRODUCT.md` 中的编号行为不变量，而不是重新陈述它们；每个重要的不变量应映射到具体的测试或验证步骤。此部分是验证存在的地方——`PRODUCT.md` 故意没有验证部分。

可选部分 — 仅在增加信号时包含。如果为空，完全省略标题；不要写"无"作为占位符。
- **端到端流程** — 仅在追踪系统路径能告诉您建议更改列表所不能告诉您的事情时包含。
- **图表** — 仅在视觉图能比散文更快地解释设计时包含 Mermaid 图表（数据流、状态转换、跨层序列）。优先使用一两个聚焦的图表，而非装饰性图表。
- **风险和缓解措施** — 仅在存在真实的故障模式、回归、迁移担忧或值得指出的推出危险时包含。
- **并行化** — 仅在工作可以清晰地拆分为多个代理，且该拆分不明显时包含。
- **后续工作** — 仅在存在推迟的清理或未来工作值得命名时包含。

## 长度启发式

将规格与功能大小匹配：
- 单文件更改，方法清晰：跳过技术规格或保持在约 40 行以内。
- 多模块更改，有一些模糊性：目标约 80–150 行。
- 大型跨领域或架构新颖的更改：只要每个部分都有其价值，更长也可以。

如果上下文和建议的更改最终从不同角度描述了相同的文件和状态，合并它们。

## 编写指导

- 将计划建立在真实的代码库结构和模式上。
- 优先使用具体的实现指导，而非通用的架构语言。
- 解释为什么建议的设计适合此仓库。
- 引用 `PRODUCT.md` 以获取行为，而不是重新陈述它。
- 每个部分都应该有其价值——如果一个部分会重复另一个部分或只包含样板，省略它。

## 保持规格最新

已批准的规格可以在同一 PR 中与实现一起发布。当模块边界、实现顺序、风险、验证策略或推出假设发生变化时，在同一 PR 中更新 `TECH.md`。签入的规格应描述实际发布的实现。

对于大型功能，实现者可以选择保留一个 `DECISIONS.md` 文件，总结具体决策。如果它会帮助未来的代理，请主动提供；否则跳过。
