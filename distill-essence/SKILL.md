# distill-essence — 快速精华萃取 × 第二大脑 × 元认知训练

> 一句话：用最少的**时间**，把一个新事物里**可复用的精华**（概念、模式、决策智慧、反模式）萃取出来，沉淀成**可被 AI 检索的第二大脑条目**，同时用元认知自检对抗「自我感动式学习」。

## 这个技能是什么 / 不是什么

**是**：一个「精华萃取器」。输入任何新事物（README、官方文档、设计文档、AI 总结、核心代码片段、业务资料），输出 1-2 张高质量、可检索、带元认知自检的「第二大脑条目」。

**不是**：
- 不是逐行读源码的「深度学习」——那要大把时间，留给 `deep-learn-codebase`
- 不是抄笔记 / 搬运文档——那是自我感动，是生产力陷阱
- 不是替代动手——萃取的终点是「带着认知去验证」，不是「读完就完」

## 三条核心理念

1. **萃取而非通读**：你不需要懂全部，你需要的是**能复用的精华**——概念、模式、决策智慧、边界。80/20。
2. **第二大脑而非一次性文档**：产出必须**可检索、可被 AI 引用**（frontmatter + 索引）。写完找不到 = 不存在。
3. **元认知是底层引擎**：判断「我真的懂了吗 / 我哪里其实在猜」，比多学一个概念更重要。元认知缺失 = 自我感动。

## 执行者说明

本技能由**主 agent** 直接执行（轻量流程，通常无需并行子代理）。
- 步骤 1 的目标对齐，用 `AskUserQuestion` 工具与用户交互
- 其余步骤主 agent 直接做；文件查找用 `Glob`，命令行检索用 `Bash` 执行 `grep`/`find`
- 读核心代码骨架时，若项目有 codegraph 索引优先用 `codegraph_explore`（一次覆盖多文件），无则 `Glob`+`Read`

## 工作流程（4 步，轻量）

### 步骤 1｜对齐 + 元认知基线

用 `AskUserQuestion` 确认：
- **学什么**：目标事物是什么？（一个框架 / 一项技术 / 一个项目 / 一个业务领域 / 一个概念）
- **为什么**：它要解决你什么问题？**必答**——没有目标的萃取会退化为好奇心驱动的自我感动
- **时间预算**：这次萃取打算花多久？（以小时计，追求性价比）
- **产出位置**：第二大脑条目写到哪？（默认 `./second-brain/`；跨项目的个人知识库建议 `~/second-brain/`）
- **元认知基线**：动手前先自问——「我现在对这个东西的理解，自评几成？我以为什么懂、其实可能不懂的地方在哪？」记下来。萃取后回看，这是元认知训练的第一步。

### 步骤 2｜精华采集（不逐行读源码）

多源快速采集，**广度优先，止于够用**：
- 优先读：`README` / 官方「核心概念」文档 / 设计文档 / 架构说明 / 优秀的 AI 总结
- 抽看：核心抽象的接口 / 入口（用 `codegraph_explore` 或 `Glob`+`Read`，只看骨架，不看实现细节）
- 必要时补：关键 issue / 作者博客 / 论文摘要
- **熔断**：在某一来源上耗时许久仍没提炼出「可复用精华」，立即换源，不死磕

### 步骤 3｜蒸馏（提炼可复用的精华）

从采集到的材料里，提炼这几类**抓手**（能复用的才算数）：
- **核心概念**：它定义了什么关键抽象？（一句话能说清）
- **模式与惯用法**：它怎么解决一类问题？这个解法能迁移到别处吗？
- **决策智慧**：作者为什么这么设计？**必须区分「有依据的」和「推测的」**（见元认知模块的知识边界地图）
- **反模式与边界**：什么场景下它会失效 / 是坑？——这一项最值钱，也最容易被略过
- **与你已知事物的关系**：它和你熟悉的 X，本质相同还是不同？（迁移点）

### 步骤 4｜产出第二大脑条目 + 元认知自检

- 按 `references/templates.md` 的「第二大脑条目模板」写条目（带 frontmatter，可检索）
- 更新 `second-brain/index.json`（机器可读索引）
- **过元认知自检**（不可跳过）：「懂了吗」自检协议 + 知识边界地图（见下方元认知模块）
- 萃取完，回答自己一个问题：**「这些认知，我打算在哪个真实项目 / 问题上用它验证？」**（防止萃取变成拖延）

## 元认知训练模块（核心，独立训练）

> 元认知 = 对自己认知过程的认知。技术敏感度、业务理解、避免自我感动，底层都靠它。

详见 `references/templates.md`，这里列四个可操作抓手 + 一份偏见清单：

1. **「懂了吗」自检协议**（四关，答不上 = 没懂）：费曼关 / 边界关 / 反事实关 / 迁移关
2. **「知识边界地图」**：每条萃取显式分三栏——**我确定的 / 我推测的 / 我不知道的**。这是对抗「编造的确定感」和达克效应的核心。
3. **「认知修正日志」**：记下「我以为 X，后来发现是 Y」。攒久了，你会看见自己的认知模式 / 偏见。
4. **「提问质量自检」**：你提的问题，是触及本质，还是浮在表面？
5. **程序员常见认知偏见清单**：达克效应、确认偏误、可得性偏差、新玩具效应、自我感动陷阱——每条配「怎么破」。

元认知既能随每次萃取训练，也能独立做周复盘（见 `references/templates.md` 第十节）。

## 判读框架（两个抓手）

- **技术敏感度 → 新技术四问**：①它解决什么**真**问题（还是伪需求）？②它替代了什么？③它的**代价**是什么？④它在 hype 周期的哪个阶段？
- **业务理解 → 五锚点**：①核心实体 ②核心流程 ③核心不变式 ④核心约束 / 规则 ⑤核心风险 / 边界。

## 反陷阱护栏（硬约束，不可跳过）

这几条是把「自我感动式学习」的陷阱内化成技能护栏：

1. 每条萃取必须过「懂了吗」至少 **2 关**，否则标注 `status: 未消化`，**不进第二大脑索引**
2. 必须有「知识边界地图」（区分确定 / 推测 / 不知道）——**没有「我推测的」标注的决策智慧，默认不可信**
3. 必须有「可迁移抓手」——萃取不能止步于「我懂了这个东西」，要回答「它怎么用到我的问题」
4. 第二大脑条目必须可检索（frontmatter + `index.json` 齐全）——否则不收录
5. 萃取后必须有「去哪验证」的出口——萃取不是终点，动手验证才是

## 详细模板与规范

第二大脑条目模板（含 frontmatter 字段定义）、元认知四个抓手的详细写法与示例、认知偏见清单、两个判读框架的详解、元认知独立训练法——全部在 `references/templates.md`。
