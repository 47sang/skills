## 一、第二大脑条目模板（核心产出物）

每条萃取产出一个 Markdown 文件，**必须带 frontmatter**（可被 AI 工具检索 / RAG 引用）。

### frontmatter 字段

```yaml
---
title: 条目标题（动宾结构，如「虚拟线程：轻量并发的新范式」）
type: concept | pattern | decision | anti-pattern | domain-model | tech-radar
tags: [并发, jvm, java21]
source: 来源（项目地址 / 文档 URL / 书名）
source_type: doc | source-code | ai-summary | business-material | paper | practice
key_concepts: [VirtualThread, 结构化并发, 线程锚定]
confidence: high | medium | low   # 这条萃取整体的可信度
status: draft | verified | unabsorbed | superseded
extracted_at: 2026-06-18
relations: [并发模型对比, reactor-vs-thread]   # 关联的其他条目名
---
```

### 正文结构

```markdown
> 一句话定位：本条讲什么、读完能用它做什么。

## 核心概念
- 是什么 / 为什么需要 / 解决什么问题
- 善用读者已知的东西做类比

## 可复用模式
- 它怎么解决一类问题？这个解法能迁移到哪？

## 决策智慧（带边界标注）
- 作者为什么这么设计？
- ⚠️ 推测 / ✅ 有依据 —— 见文末「知识边界地图」

## 反模式与边界（最值钱）
- 什么场景下它会失效 / 是坑？给具体可复现的故障场景

## 与已知事物的关系（迁移点）
- 它和你熟悉的 X，本质相同还是不同？

## 元认知自检
- 「懂了吗」四关：……
- 知识边界地图：……

## 去哪验证
- 我打算在哪个真实项目 / 问题上用它验证？
```

### 示例条目（虚拟线程）

```markdown
---
title: 虚拟线程：轻量并发的新范式
type: concept
tags: [并发, jvm, java21]
source: JEP 444
source_type: doc
key_concepts: [VirtualThread, 结构化并发, 线程锚定]
confidence: high
status: verified
extracted_at: 2026-06-18
relations: [并发模型对比]
---
> 虚拟线程让你用「同步阻塞写法」拿到「异步高并发」的效果，代价是少数隐式锚定（pinning）陷阱。

## 核心概念
- 是什么：由 JVM 调度、而非操作系统调度的轻量线程；阻塞时让出载体线程，而非占用一个 OS 线程。
- 为什么需要：平台线程贵（一个几 MB 栈 + 一个 OS 线程），虚拟线程便宜（可轻松开百万级）。
- 解决什么：让「一个请求一个线程」的直观模型重新可用，免去了 reactive 编程的回调地狱。
- 类比：平台线程像「专车」（一客一车，贵），虚拟线程像「共享单车」（随用随还，调度器复用载体线程这片「停车场」）。

## 可复用模式
- 「同步写法 + 运行时调度复用」：当你看到一个 API 提供「看起来阻塞、实则让出资源」的抽象（虚拟线程、goroutine、async/await），本质都是这个模式。判断一个并发抽象好不好用，看它有没有保留「同步代码的直观性」。

## 决策智慧（带边界标注）
- ✅ 为什么让 JVM 而非 OS 调度：为了在阻塞点（IO）透明地让出载体线程，复用成本极低。——有 JEP 设计文档支撑。
- ⚠️ 为什么不直接改造 synchronized 而是逐步废弃：synchronized 持有监视器时会锚定载体线程，JDK 21 只改造了部分。推测：完全去除锚定要重写监视器实现，工程量与兼容性代价大，故分期。

## 反模式与边界（最值钱）
- 🔴 在虚拟线程里用 `synchronized` 持有监视器并阻塞 → 载体线程被锚定，并发度崩塌。改用 `ReentrantLock`。
- 🔴 把虚拟线程和「线程局部大对象 + 池化」混用 → 池化假设线程少，虚拟线程数量爆炸会撑爆。
- 🟡 虚拟线程不省 CPU——它解决的是「等待」（IO 密集），不是「计算」（CPU 密集）。CPU 密集任务仍要靠平台线程池控制并行度。

## 与已知事物的关系（迁移点）
- 和 Go 的 goroutine：本质相同（用户态调度 + 阻塞让出）。差异：虚拟线程强调「让现有同步代码零成本迁移」，goroutine 是新语言从零设计。
- 和 reactive（WebFlux）：目标相同（高并发低资源），手段相反——reactive 要你改写法（回调 / Flux），虚拟线程让你保持同步写法。

## 元认知自检
- 费曼关：已用「共享单车 vs 专车」给外行讲清。✅
- 边界关：CPU 密集任务不适用；synchronized 锚定。✅
- 反事实关：若没有虚拟线程，高并发只能上 reactive，代码可读性下降。✅
- 迁移关：可迁移到判断「任何『阻塞即让出』的并发抽象」。✅
- 知识边界地图：
  - ✅ 确定：调度模型、synchronized 锚定、不省 CPU
  - ⚠️ 推测：synchronized 改造分期原因
  - ❓ 不知道：载体线程池的默认大小策略细节

## 去哪验证
- 在转 Agent 的项目里，把一个并发调用多个工具的节点，用虚拟线程 + 结构化并发重写，对比 Future/CompletableFuture 的写法与可读性。
```

## 三、「懂了吗」自检协议（元认知抓手 1）

萃取完一个概念，过四关。**任何一关卡住，就是没真懂，标 `status: unabsorbed`。**

| 关 | 问题 | 答不上意味着 |
|---|---|---|
| **费曼关** | 能用大白话给一个外行讲明白吗？（不用术语） | 你只是记住了术语，没理解本质 |
| **边界关** | 它在什么场景下会**失效 / 不适用**？ | 你只知道它好的时候，不知道它的代价——典型的达克效应 |
| **反事实关** | 如果它**不存在 / 换一种实现**，会怎样？ | 你没理解它解决了什么真问题 |
| **迁移关** | 它和你**已知**的某个东西，本质相同还是不同？ | 它孤立地存在你脑子里，无法复用——萃取失败 |

**用法**：在条目的「元认知自检」章节，逐关写一句话答案。过 2 关以上才能进索引；过 4 关是「真正消化」。

## 四、知识边界地图（元认知抓手 2，最重要）

对每条萃取（尤其是「决策智慧」），显式分三栏。**这是对抗「编造的确定感」的核心。**

```markdown
## 知识边界地图
- ✅ **确定的**（有源码 / 文档 / 实测支撑）：……
- ⚠️ **推测的**（基于经验 / 训练数据的合理猜测）：……
- ❓ **不知道的**（明确自己的盲区）：……
```

**为什么重要**：
- 不区分这三栏，你会把「AI / 大脑顺嘴说的合理推测」当成事实——这就是上一轮批评的「决策复盘退化成编造」。
- 写下「❓ 不知道的」，逼你承认盲区——这是达克效应的解药。
- 写下「⚠️ 推测的」，将来验证时你知道哪些要先打问号。

## 五、认知修正日志（元认知抓手 3）

单独维护一个 `second-brain/corrections.md`，每条记录一次「我以为 → 其实」。

```markdown
## 2026-06-18｜虚拟线程
- **我以为**：虚拟线程省 CPU，所以 CPU 密集任务也该用它。
- **其实是**：它只省「等待」（IO），CPU 密集任务该用平台线程池控并行度。
- **为什么会错**：把「轻量」等同于「万能」，没区分瓶颈类型。
- **教训**：评估任何「更快 / 更轻」的技术，先问「它优化的是哪个维度，代价压到哪个维度」。
```

**为什么重要**：单条修正没什么，**攒 20 条之后**，你会看见自己反复犯的认知模式（比如「总是高估新工具的适用范围」）。看见模式，才能改模式。这是元认知的高级形态。

## 六、提问质量自检（元认知抓手 4）

萃取 / 学习时，问自己：我提的问题是好问题还是烂问题？

- **烂问题**（浮在表面）：「这个 API 怎么用？」「它支持哪些参数？」——答案查文档就有，问完没长进。
- **好问题**（触及本质）：「它为什么选这个设计而不是那个？」「它在什么场景下会坑我？」「它和我已知的 X 哪里本质不同？」「如果我自己造一个，会卡在哪？」

**判据**：好问题的答案，会**改变你后续的行动 / 判断**；烂问题的答案，只是多了一条可查的信息。

**用法**：每次萃取前，先列 3 个你最想搞懂的问题。萃取后检查——这 3 个问题是不是「好问题」？被回答了吗？没被回答的要继续追，别放过。

## 七、程序员常见认知偏见清单（每条配「怎么破」）

| 偏见 | 表现 | 怎么破 |
|---|---|---|
| **达克效应** | 能力不足时反而高估自己（因为看不到自己的无知）；真懂了反而会低估 | 用「边界关」+ 知识边界地图，强制写下「❓ 不知道的」 |
| **确认偏误** | 只找支持自己判断的证据，忽略反例 | 萃取时强制写「反模式与边界」——主动找它失效的场景 |
| **可得性偏差** | 容易想到的（最近看的、最热门的）被当成最重要的 | 萃取前问「这玩意是真重要，还是因为它最近刷屏」——用技术敏感度四问的 hype 阶段校准 |
| **新玩具效应** | 追新框架 / 新工具带来的多巴胺，误当成「学习进步」 | 每学一个新东西，过「迁移关」——它能不能用到我现有问题？用不上就是新玩具 |
| **自我感动陷阱** | 把「整理笔记 / 产出文档」当成「学会了」 | 每条萃取必须过「懂了吗」四关 + 有「去哪验证」出口；产出文档 ≠ 获得能力 |
| **锚定效应** | 被第一个看到的解释 / 数字锚住，后续判断偏不回来 | 刻意找 2 个不同来源、甚至对立观点的资料，对比其差异点 |

## 八、技术敏感度四问（判读框架 1）

面对任何新技术 / 新框架，快速回答四问。**答不出「解决什么真问题」和「代价」，就是没看懂。**

1. **它解决什么真问题？**（还是为了用而用 / 伪需求？）
   - 破：如果这个问题不存在，这门技术还有意义吗？
2. **它替代了什么？**（手写的什么、上一个方案的什么）
   - 破：它取代的那个东西，在什么场景下其实还更好？
3. **它的代价是什么？**（没有银弹）
   - 破：它把复杂度 / 风险转移到了哪里？（通常是「调试更难」「心智负担更重」「生态不成熟」之一）
4. **它在 hype 周期的哪个阶段？**
   - 触发期（炒作猛、坑多）→ 期望膨胀顶点 → 幻灭低谷（弃用潮）→ 稳步爬升（真有用）→ 生产成熟
   - 破：在哪个阶段投入精力最划算？通常「幻灭低谷之后」性价比最高

**示例（对一个新出的 Agent 框架）**：
- 真问题：把多步 LLM 调用编排成可恢复、可观测的状态机——✅ 真问题。
- 替代：手写 while 循环 + 全局变量管理对话状态。
- 代价：学一套状态图 DSL；调试要在「非确定性输出」上排查，比传统代码难。
- hype 阶段：尚在期望膨胀期——可用，但别赌它一年后 API 不变。

## 九、业务理解五锚点（判读框架 2）

面对任何业务系统 / 业务领域，快速抓这 5 个锚点。**抓不全就还没理解这个业务。**

1. **核心实体**：这个业务在「处理」什么？（电商=订单/商品/用户；保险=保单/被保人/理赔）
2. **核心流程**：这些实体怎么流转？（下单→支付→发货→收货；投保→核保→出单→报案→理赔→结案）
3. **核心不变式**：什么必须永远成立？（账户余额≥0；库存不能为负；理赔金额不超过保额）
4. **核心约束 / 规则**：业务上必须遵守的规则？（监管要求；风控阈值；业务边界）
5. **核心风险 / 边界**：哪里最容易出事、最需要防？（资金对账；并发超卖；欺诈；合规）

**用法**：读一个新项目 / 新业务，第一件事就是填这 5 个锚点。填完，你就有了「这个业务的骨架」，后续所有细节都能挂上去，不会迷失在实现里。

## 十、元认知独立训练（不依赖具体萃取）

元认知不只是萃取时用，可以独立练。建议每周做一次复盘（5-10 分钟）：

```markdown
## 本周元认知复盘（YYYY-MM-DD）
1. **本周我「以为懂、其实不懂」的有哪些？**（抓达克效应）
2. **本周我做了哪些「自我感动」的事？**（产出很忙，但能力没长）
3. **本周我的哪个判断被现实打脸了？为什么？**（认知修正）
4. **下周我打算怎么验证本周学到的东西？**（防止准备性拖延）
```

**关键**：第 4 条是出口闸门。如果连着几周第 4 条都答不上（学了但没去用），说明你正在用「学习」逃避「动手」——这是陷阱⑤的信号，立刻停手去写代码。

## 关键认知（写给执行这个技能的你）

1. **萃取的终点是「能用」，不是「产出文档」**。如果一条萃取没有「去哪验证」的出口，它就是自我感动。
2. **知识边界地图比内容本身更重要**。诚实地写下「我不知道」，比写满一篇漂亮但真假难辨的条目有价值。
3. **第二大脑的价值在「复利」**：单条没什么，100 条可检索、带边界的条目，会让你越来越快地理解新事物——因为新事物大多是旧模式的变体。
4. **元认知是这一切的底层引擎**：你越能判断「我真的懂吗 / 我哪里在猜」，你的技术敏感度、业务理解、学习能力就涨得越快。这不是软技能，是你转型最稀缺的硬通货。
