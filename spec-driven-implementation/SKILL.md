# spec-driven-implementation

驱动重要功能的规格优先工作流程。

## 概述

对于书面规格能提高实现质量、减少模糊性或使评审更容易的重要功能使用此技能。务实至上：并非每个更改都需要规格。

规格通常应位于：
- `specs/01-门诊混发/PRODUCT.md`
- `specs/01-门诊混发/TECH.md`

例如：
- `specs/01-门诊混发/PRODUCT.md`
- `specs/01-门诊混发/TECH.md`

`specs/` 应仅包含以编号-描述命名的目录作为直接子级。不要在那里创建工程师命名的子目录或功能 slug 目录。

如果尚不存在相关的 Linear issue，请在编写规格之前创建一个。直接使用 Linear MCP 工具：
- `list_teams` 找到合适的团队
- `list_issue_labels` 检查预期的标签/标记
- `save_issue` 使用合适的团队和标签创建 issue

如果请求和周围上下文中团队或标签不明显，使用 `ask_user_question` 澄清，而不是猜测。

这些规格应主要由代理编写，而非手写，并应签入源代码控制，以便它们可以被评审并与代码保持同步。

## 何时需要规格

在以下情况下强烈推荐使用规格：
- 产品或架构模糊性
- 预期实现规模约为 1k+ 行代码
- 深度或跨领域的堆栈更改
- 回归成本高昂的风险行为更改
- 代理质量将从更清晰的输入中显著改善的工作

规格通常对以下情况不必要：
- 小型、本地 bug 修复
- 简单的重构
- 几乎没有模糊性的狭窄 UI 调整

对于纯 UI 更改，产品规格通常有用，而技术规格可能不必要。

## 工作流程

### 1. 决定功能是否需要规格

评估功能的大小、模糊性和风险。如果规格不会显著改善执行或评审，跳过它们，专注于验证。

### 2. 确定 specs 目录编号并创建目录

在编写规格之前，确定 specs 目录的编号并确保目录存在。

**目录命名格式：** `NN-描述`，其中 `NN` 是两位数字序号（01-99），`描述` 是短横线连接的中文功能名称。例如：`01-门诊混发`、`02-药品追溯`。

**编号规则：**
- 查看 `specs/` 下已有的目录，提取最大的两位数字序号。
- 如果 `specs/` 为空或不存在，从 `01` 开始。
- 新目录编号 = 现有最大编号 + 1。
- 如果计算结果超过 99，停止并请用户介入，不要继续自动递增。

**创建规则：**
- 如果对应目录已存在，直接复用，不创建新目录。
- 如果对应目录不存在，直接创建目录及后续的 `PRODUCT.md`、`TECH.md` 文件占位。
- 目录创建后，后续的产品规格和技术规格都应写入该目录。

### 3. 先编写产品规格

在实现之前，创建 `PRODUCT.md` 描述期望的用户可见行为。

使用 `write-product-spec` 技能生成它。产品规格应定义：
- 正在解决什么问题
- 期望的用户体验
- 不变量和边缘情况
- 成功标准
- 行为将如何验证

如果功能有 UI 或交互设计，询问用户是否存在 Figma 模型。如果没有模型，继续但在产品规格中显式指出这一点。

引用规格中的 Linear issue（如果存在）。因为规格位于 `specs/01-门诊混发/...` 下，这通常应该是直截了当的。

### 4. 在需要时编写技术规格

对于实质性或模糊的实现工作，使用 `write-tech-spec` 技能。

在以下情况下优先使用技术规格：
- 实现跨越多个子系统
- 架构或可扩展性很重要
- 有值得记录的有意义权衡
- 评审者从审查计划而非原始代码中受益更多

如果在端到端原型之后编写技术规格导致更准确的实现计划，则这是可接受的。当实现细节仍然太不确定时，不要强迫编写过早的技术规格。

### 5. 实现已批准的规格

规格获得批准后，使用 `implement-specs` 技能从已批准的 `PRODUCT.md` 和 `TECH.md` 构建。

实现通常可以与产品和技术规格一起推送到同一个 PR 中。随着工程师的迭代，保持 `PRODUCT.md`、`TECH.md`、代码更改和测试在同一个 PR 中，以便评审反映实际将发布的功能。

对于大型功能，实现者可以选择提供：
- `PROJECT_LOG.md` 以跟踪探索的路径、检查点和当前实现状态
- `DECISIONS.md` 以捕获设计和实现过程中做出的具体产品和技术决策

这些是可选的辅助工具，不是必需的输出。

### 6. 在实现过程中保持规格最新

如果实现与规格不同，更新规格，而不是让它过时。

在以下情况下更新 `PRODUCT.md`：
- 用户可见行为发生变化
- 成功标准发生变化
- UX 细节或边缘情况发生变化

在以下情况下更新 `TECH.md`：
- 实现方法发生变化
- 架构边界移动
- 风险、依赖项或推出细节发生变化
- 测试或验证计划发生变化

签入的规格应描述实际发布的功能，而不仅仅是初始意图。只要实际可行，将这些规格更新保持在与其相关代码更改相同的 PR 中。

### 7. 根据规格验证行为

在认为工作完成之前，确保验证映射回规格。优先使用直接验证产品行为的测试和工件：
- 对 crate 级别的单元测试和回归覆盖使用 `rust-unit-tests` 技能
- 对关键用户流程使用集成测试
- 在适当时使用 loom 演练或等效的功能演示
- 对 UI 繁重的工作，截图或视频有用时使用

## 最佳实践

- 务实至上。
- 编写规格以改善代理的输入质量，而非作为仪式。
- 保持产品规格以行为为导向，实现细节轻量。
- 保持技术规格以实现为导向，并基于当前代码库模式。
- 利用评审时间验证规格和行为，而不是过度关注代码风格细节。
