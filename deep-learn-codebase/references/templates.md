# deep-learn-codebase — 详细模板与规范

本文件是 [SKILL.md](../SKILL.md) 的配套参考，提供可直接复用的模板、规范、prompt 骨架。

---

## 一、教学文档结构模板（每篇统一）

每篇文档按此结构写（顺序与标题可微调，但要素要齐）：

```markdown
# 标题：主题名

> 一句话定位：本篇讲什么、读者读完能掌握什么。

## 🎯 本篇你将学到
- 3–6 条要点

## 🧠 核心概念
- 通俗讲解「是什么、为什么需要、解决什么问题」
- 善用读者已知的东西做类比（如 Java 经验类比 TS 概念）

## 🔍 源码剖析
### 子主题 1
- `文件相对路径:行号` 引用
- 精简代码片段（只留关键行 + 中文注释）
- 讲解
（分若干小节）

## 📐 图解
- Mermaid 图（flowchart / sequenceDiagram / classDiagram），准确反映真实结构

## 🎓 设计精妙之处与可借鉴点
- 每点提炼「为什么这么设计」+「可借鉴到我自己的框架」

## ⚠️ 常见坑与注意事项

## 🏋️ 刻意练习：缺陷预演        ← 中层训练（见模板二）
## 🎯 决策复盘：复现作者的取舍    ← 深层训练（见模板三，仅核心篇）

## 🔗 与其他模块的关系
- 交叉引用同目录其他文档名

## 📚 小结
```

**风格要求**：简体中文；技术标识符（类名/方法/路径/注解）保留英文放反引号；老师口吻由浅入深；emoji 仅在标题/重点适度用；篇幅 2000–4000 字。

## 二、缺陷预演练习模板（中层）

聚焦**一个设计**：「不用它会怎样 / 用错会怎样」。每个设计写一个练习块：

```markdown
### 练习 N｜设计名

- 🔴 **反模式预演**：如果不用这个设计（改用某替代方案），会埋什么雷？给具体场景，逼读者预判代价。
- 🟡 **陷阱预判**：这个设计被误用、或在错误场景使用，会怎样？（有料才写，可选）
- 💡 **参考思路**：一句话点破本质。这是「先让读者自己想」的参考答案，不是正文复述。
```

**设计要求**：
- 必须基于真实源码，引用 `文件:行号`
- 要「扎心」——构造具体攻击/故障场景，不是软绵绵复述好处
- 参考思路延迟给出（鼓励读者先想 2 分钟）
- 每篇 2–3 个，不堆砌

**示例**（Java final 模板方法）：
> 🔴 反模式预演：如果 `call()` 不是 final 而是要求子类「记得调 super」，会埋什么雷？
> 💡 参考思路：final 把「不变式保证」从「靠人自觉」变成「靠编译器强制」。凡是「绝对要保证每次都发生」的逻辑用 final；凡是「允许子类定制」的留 abstract 钩子。

## 三、决策复盘模板（深层，仅核心篇）

聚焦**一个岔路口**：面对问题有多个方案，为什么选这个、放弃什么、边界在哪。区别于缺陷预演——它复现「取舍思考过程」。

```markdown
### 决策 N｜决策名（一句话点出岔路口）

**场景**：一两句描述作者面临的决策问题。

- 方案 A：……（一句话做法）
- 方案 B：……（必要时方案 C）

**你来权衡**：各方案的优势/代价？在什么条件下选择会反转？

- 💡 **参考思路**：三要素缺一不可——
  ① 作者实际选了什么；
  ② 为什么（换来什么）；
  ③ 这个决策的边界（什么场景下会后悔/失效）。
```

**设计要求**：
- 每个方案的代价必须真实、沉重，**不能是稻草人**
- 必须包含「边界/反转条件」——什么场景下这个选择会失效
- 规避与升华篇已有的宏观决策重复，聚焦本篇细节
- 核心篇（灵魂篇）2–3 个，由你亲自写或严格审阅

**示例片段**（循环实现）：
> 方案 A：while + block()；方案 B：flatMap 递归；方案 C：显式状态机。
> 💡 参考思路：A 的 block 会撕碎 Mono.using 的资源释放契约；C 的状态机对两态循环样板超过收益。作者选 B。边界：若团队只有 2 人、从不并发、模型调用极快，A 的直观性压倒 B——但 agent 场景天生「工具慢、并发高」，所以 A 几乎永远不是正解。

## 四、Workflow agent prompt 骨架（阶段 3 直接复用）

阶段 3 并行深挖时，给每个 agent 的 prompt 按此结构（替换 `{主题}`/`{文件}`/`{范围}`）：

```
你是 {项目名} 的资深架构导师，为「精通 {读者熟悉的生态}、但不熟本框架、想理解核心思想」的学习者撰写教学文档。

仓库根：{ROOT}
输出文件（用 Write 写入）：{OUT}/{文件名}

【必须深入分析的源码范围】{该主题涉及的包/类/文件}
【读者读完要掌握】{该主题的核心目标}
【重点剖析方向】{3–6 条该主题的关键点}

【撰写要求——严格遵守】
1. 简体中文；技术标识符保留英文放反引号。
2. 善用读者已知生态做类比。
3. 老师口吻，由浅入深：先「是什么、为什么」，再深入源码。
4. 严禁编造。每个论断基于真实源码，优先 codegraph_explore 核实。引用用「文件相对路径:行号」。
5. 至少 1–2 个 Mermaid 图，准确反映真实结构。
6. 每个设计点提炼「为什么这么设计」+「可借鉴点」。
7. 文末列「常见坑与注意事项」。
8. 文档结构统一为：{贴「教学文档结构模板」的前 7 个章节}。
9. emoji 适度；篇幅 2000–4000 字。

【全套文档清单】（供「与其他模块的关系」交叉引用，引用时写文件名）
{列出所有文档名}

【返回值】文档正文用 Write 写入文件，不要放返回值。结构化返回只含：file、title、keyPoints、sourceFilesRead、mermaidDiagrams。
```

## 五、对抗性审阅清单（阶段 4 用）

agent 产出后，逐项过：

- [ ] **完整性**：计划的主题文档是否都产出了？
- [ ] **术语一致**：grep 关键术语（如「代理」「智能体」），检查是否有误用/混用
- [ ] **交叉引用**：被引用的文件名是否都真实存在（`grep -ohE '文件名' *.md | sort -u` 逐个 check）
- [ ] **Mermaid 数量**：每篇是否都有图、数量是否合理统一
- [ ] **技术点栽样核实**：随机挑 5–8 个 agent 提到的关键类/方法/字段，`find`/`grep` 确认真实存在（重点抓编造的类名、错误的行号、从旧文档沿用的过时叫法）
- [ ] **抽读全文**：1–2 篇通读，确认深度、准确、「扎心度」
- [ ] **修正**：发现错误立即改所有引用处（如某类不存在，全仓库替换为正确叫法）

**核心原则**：agent 用了真实源码 ≠ agent 没编造细节。验证而非盲信。

## 七、关键认知（写给执行这个 skill 的你）

1. **亲自写灵魂，委托挖细节**：核心抽象/主循环/设计思想必须你亲自写；子系统细节可委托 agent 并行。
2. **先做样板再并行**：练习层先亲自写一篇做质量基准，再让 agent 参照并行。
3. **判断力 ≠ 阅读**：交付时务必告诉用户「练习要自己想，参考思路延迟看」，否则文档白做。
4. **方法论语言无关**：6 阶段流程对任意语言/项目通用，只是工具不同（Java 用 codegraph，JS 用 ts-morph/grep，等等）。
