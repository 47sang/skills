# 深度学习代码库

把一个陌生代码库变成一套「老师视角」的教学文档 + 刻意练习，让学习者从「看懂」走到「会判断」。

## 两条铁律（贯穿全程）

### 铁律一：三层判断力

| 层 | 内容 | 怎么训练 |
|---|---|---|
| 表层 | 语法 / API | 查文档、问大模型（产出里点到为止，价值有限） |
| 中层 | 模式 / 惯用法 | 逆向拆解 + **缺陷预演**（不用它会怎样 / 用错会怎样） |
| 深层 | 取舍 / 代价 | **技术决策复盘**（假设你是作者，列方案权衡、对照实际选择） |

文档不能只讲「为什么这么设计」（中层拆解），必须配套**缺陷预演**（中层）和**决策复盘**（深层）。大模型解决了表层，中深层只能靠刻意练习。

### 铁律二：文档是地图，不是肌肉

被动阅读无法获得判断力。文档提供「拆解的素材」和「练习的题目」，但「想」这个动作只能学习者自己做。练习要「扎心」——逼读者预判代价，参考思路**延迟给出**（鼓励先自己想 2 分钟）。

## 工作流程（6 阶段）

### 阶段 0：确认目标（必做，别跳过）

用 `AskUserQuestion` 或对话确认：

- **学习目标**：整体架构？某子系统？某技术点？
- **深度**：概览（3–5 篇）/ 标准（8–12 篇）/ 深度（15+ 篇）
- **产出位置**：默认 `docs-study/`（先检查是否已有 `docs/`，避免大小写冲突或污染官方文档）
- **读者背景**：熟悉什么语言/生态（决定类比方式——用读者已知的东西类比未知）

> 宁可花 5 分钟确认，也别花数小时做出错方向。

### 阶段 1：地形侦察

摸清地形，建立鸟瞰：

- 项目结构（`codegraph_files` / `find` / `ls`）
- 构建工具与依赖（`pom.xml` / `package.json` / `build.gradle` / `Cargo.toml` / `go.mod`…）
- 技术栈、语言版本、关键约束（读 `CLAUDE.md` / `README` / `CONTRIBUTING`）
- 找**入口**（`main` / 核心接口 / 路由 / 主循环）和**「灵魂」候选**（被依赖最多的核心抽象）
- **优雅降级**：有 `codegraph` 索引优先用 `codegraph_explore`（一次覆盖多文件源码，最高效）；没有就用 `grep` / `Glob` / `Read`

### 阶段 2：吃透核心（亲自，不委托）

这是灵魂，**不能全交给 agent**：

- 逐行读核心抽象：主接口、基类、主循环、主流程
- 用 `codegraph_node` 拿单个方法/类的完整源码（含调用链 trail）
- 建立心智模型：数据怎么流、控制怎么转、关键不变式是什么
- 产出：你心里要先有一张「这个系统靠什么运转」的清晰图，才能指挥 agent

### 阶段 3：主题切分 + Workflow 并行深挖

把项目切成 N 个主题（按模块 / 子系统 / 核心抽象切），每个 agent 深入一个主题产出一篇教学文档：

- 用 **Workflow 工具并行**（每个主题一个 agent）
- 给 agent **严格统一的 prompt**：文档结构模板、风格、必须读真实源码、全套文档清单（供交叉引用）、schema 约束返回值
- agent 必须基于真实源码（**不准编造**），优先 `codegraph_explore` 核实
- 统一文档结构（见 [references/templates.md](references/templates.md) 的「教学文档结构模板」）
- prompt 骨架直接复用 [references/templates.md](references/templates.md) 的「Workflow agent prompt 骨架」

### 阶段 4：对抗性审阅（不盲信 agent）

**agent 会编造。必须验证：**

- **批量检查**：术语一致性（如「代理」vs「智能体」是否误用）、交叉引用文件名是否都存在、Mermaid 图数量是否统一
- **栽样核实**：随机挑 agent 提到的关键类 / 方法 / 字段，用 `find` / `grep` 确认真实存在（抓编造的类名/行号）
- **修正**：发现事实错误（不存在的类、过时 API、从旧文档沿用的错误叫法）立即修正**所有引用处**
- **抽读** 1–2 篇全文，确认深度、准确性、「扎心度」
- 审阅清单见 [references/templates.md](references/templates.md) 的「对抗性审阅清单」

### 阶段 5：亲自写灵魂篇 + 综合篇

- **灵魂篇**（入口 / 心智模型 / 核心抽象 / 主循环）由你**亲自写**——这是框架心脏，质量必须最高，不能委托
- **综合升华篇**（核心设计思想、可借鉴点）由你写——需要通读全部文档后综合提炼

### 阶段 6：刻意练习层（把「看懂」变「会判断」）

给核心文档嵌入两层练习（规范见 [references/templates.md](references/templates.md)）：

- **缺陷预演**（中层）：每个核心设计配 🔴 反模式预演（不用它会怎样）+ 🟡 陷阱预判（用错会怎样）+ 💡 参考思路
- **决策复盘**（深层）：给关键岔路口配「假设你是作者，方案 A/B/C，你来权衡」+ 💡 参考思路（含「选了什么 + 为什么 + 边界/后悔点」三要素）
- **执行方式**：先亲自写一篇做**质量样板** → Workflow 并行嵌入其余篇（参照样板）→ 亲自写灵魂篇的决策复盘 → 审阅核实
- 完成后用「三层覆盖度自评表」评估，诚实告诉用户每层做到几成

## 关键约束

- **优雅降级**：`codegraph` 有则用，无则 `grep` / `Glob` / `Read`
- **分阶段确认**：阶段 0 必确认目标；遇到重大分叉（如目录名冲突、深度选择）用 `AskUserQuestion`
- **格式**：Mermaid 图（流程图/时序图/类图）+ 源码 `文件路径:行号` 引用 + 中文撰写（技术标识符保留英文放反引号）
- **不盲信**：所有 agent 产出的技术论断都要对抗性核实，修正后再交付
- **语言无关**：方法论对 Java/TS/Rust/Go/Python 通用，别绑死某一门
