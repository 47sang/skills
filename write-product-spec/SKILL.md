# write-product-spec

为重要的用户-facing 功能编写 `PRODUCT.md` 规格说明，重点关注详细的行为和验证。

## 概述

产品规格应将期望的行为描述得足够明确，使代理能够正确实现并避免回归。纯粹从用户的角度描述功能——用户看到什么、做什么、体验到什么，以及必须对他们成立的不变量。不要包含实现细节（内部类型、状态布局、模块边界、数据流、算法）。

"用户"并不限于应用的最终用户。它意味着任何消费被设计表面的主体：
- 对于 UI/UX 功能：使用产品的人类。
- 对于数据模型：读取和写入该模型的代码。
- 对于 API、协议或库：该 API 的调用者——其他服务、客户端代码、插件或代理。
- 对于 CLI 工具或面向开发者的表面：调用它的开发者。

规格应从该消费者的角度描述行为：表面的形状、他们可以执行的操作、他们看到什么返回、他们可以依赖的不变量，以及必须处理的边缘情况——而不规定表面在底层如何实现。

实现细节、验证和测试计划位于配套的 `TECH.md` 中，由 `write-tech-spec` 技能生成。编写产品规格通常是一个两步过程的第一步：一旦 `PRODUCT.md` 达成一致，调用 `write-tech-spec` 为同一功能生成 `TECH.md`（或告知用户这是预期的下一步）。产品规格的编写应使技术规格可以直接从中写出。

将规格写入 `specs/01-门诊混发/PRODUCT.md`，其中 `01-门诊混发` 可以是以下之一：
- Linear 工单号（例如 `specs/01-门诊混发/PRODUCT.md`）
- GitHub issue id，带有 `gh-` 前缀（例如 `specs/gh-4567/PRODUCT.md`）
- 短横线命名的功能名称（例如 `specs/01-门诊混发/PRODUCT.md`）

`specs/` 应仅包含以编号-描述命名的目录作为直接子级——不要创建工程师命名的子目录。

- 工单/issue 引用是可选的。如果用户有项目管理工具（如 Linear、Jira）的工单或 GitHub issue，使用其 id。如果没有，请他们提供用于目录的功能名称。仅在用户明确要求时才创建新的工单或 issue；在这种情况下使用对应的 MCP 工具或 CLI（如果团队、标签或仓库不明确，使用 `ask_user_question`）。

## 编写前

仅收集你需要的上下文：目录 id（Linear 工单、GitHub issue 或功能名称）、功能摘要、目标用户、关键行为、边缘情况，以及功能将如何验证。对于缺失的上下文使用 `ask_user_question`，而不是猜测。

### Figma 模型

如果功能有任何 UI 或交互设计，在起草行为部分之前询问用户是否存在 Figma 模型，并在提供时在规格中包含链接。模型通常是视觉状态、间距和边缘情况布局的最可靠事实来源——不询问可能导致行为部分猜测设计师已经确定的意图。

- 如果用户提供链接，在简短的 `## Figma` 部分（或行为顶部附近的内联）中将其包含为 `Figma: <link>`。
- 如果用户确认没有模型，注明 `Figma: none provided`，以便缺失是显式的而非模糊的。
- 如果功能纯粹是后端（数据模型、API、没有视觉表面的 CLI），跳过问题并省略该部分。

不要悄悄放弃设计上下文；在设计通常预期的功能上，显式的"无"优于完全不提及。

## 结构

必选部分：

1. **摘要** — 1-3 句话描述功能和期望的结果。
2. **行为** — 规格的核心内容。功能如何工作的详尽英文描述，编写为可测试的不变量编号列表。见下方的"行为部分"——这是规格体现其长度的地方，其他所有内容都应保持精简以避免重复它。

可选部分 — 仅在超出核心内容增加信号时包含。如果为空，完全省略标题；不要写"无"作为占位符。
- **问题** — 仅在动机从摘要中不明显时包含。
- **目标/非目标** — 在范围模糊或有争议时包含。
- **Figma** — 在存在链接时包含，或在设计重要但没有模型时包含显式的 `Figma: none provided` 注释。完全省略非视觉功能。见上方的"Figma 模型"。
- **开放问题** — 优先在相关行为旁边使用内联 `**开放问题：** ...`。仅在存在多个值得收集的未解决问题时才包含专门部分。

不要包含验证、成功标准或测试部分。验证和测试计划位于配套的 `TECH.md`（由 `write-tech-spec` 生成）中。将行为编写为可独立测试的不变量——技术规格可以直接引用它们。

## 行为部分

行为就是规格。其他一切都是框架。

行为的目标是对功能如何工作提供完整的英文描述，详细到技术规格可以直接从中写出，而作者无需猜测或重新推导产品意图。如果读者在阅读行为后对功能在某些情况下的工作方式仍有疑问，则该部分未完成。

至少描述：
- 默认行为和快乐路径用户流程。
- 每个用户可见状态及其之间的转换。
- 用户可以提供的所有输入以及功能如何响应。
- 空状态、错误状态、加载/待处理状态和取消。
- 合理的实现者不会想到要问的边缘情况——权限被拒绝、离线、超时、状态更改之间的竞争、多个并发实例、过时或缺失的数据、中途交互中的焦点丢失、与相邻功能的交互。
- 键盘、辅助功能和焦点期望（如相关）。
- 始终成立的不变量以及不得回归的行为。

行为的长度应与功能匹配。平凡功能可能只需要少量不变量；复杂功能可能需要很多，每个流程或状态有子部分。规格的其余部分应保持精简，以便行为可以尽可能详尽，而不会使整体文档膨胀。倾向于枚举一个额外的边缘情况，而不是少一个。

## 长度启发式

行为应如功能所需一样长——不要为了达到行数目标而截断边缘情况。以下启发式适用于行为周围的所有内容（摘要、可选部分）：保持框架精简，以便规格的总长度反映功能的实际复杂性，而非结构性开销。
- 简单的修复或狭窄的 UI 调整：无需规格。
- 小功能（单个模块，少数边缘情况）：框架加行为通常总共约 30-60 行。
- 中等功能（跨模块，多个状态）：通常总共约 80-150 行。
- 大型或行为丰富的功能：更长也可以，大部分长度应存在于行为中。

如果你发现自己在摘要、问题、目标和行为中重复同样的想法，合并框架——而不是合并行为内容。

## 编写指导

- 优先使用具体的、可观察的行为，而非 aspirational 的措辞。
- 尽可能将行为编写为不变量列表，而非散文。
- 捕获不得回归的不变量和容易遗漏的的边缘情况。
- 除非 UX 需要，否则避免实现细节。
- 每个部分都应该有其价值——如果一个部分会重复另一个部分或只包含样板，省略它。

## 保持规格最新

已批准的规格可以在同一 PR 中与实现一起发布。随着实现演变，当用户可见行为或 UX 细节发生变化时，在同一 PR 中更新 `PRODUCT.md`。签入的规格应描述实际发布的功能。

对于大型功能，实现者可以选择保留一个 `DECISIONS.md` 文件，总结设计和实现过程中做出的具体决策。如果它会帮助未来的代理，请主动提供；否则跳过。

## 行为部分示例

一个假设功能的示例行为部分：在应用块列表中渲染 GitHub 风格的 Markdown 表格。它展示了预期的形状——编号、可测试、用户视角的不变量，枚举默认值、边缘情况、格式错误的输入、流式传输、选择/复制、搜索、共享、主题和跨表面一致性，并带有一个内联的开放问题。

````markdown
## 行为

1. 当终端输出块包含 GitHub 风格的 Markdown 表格（标题行、一个或多个 `---` 段的分隔行，以及一个或多个正文行，全部由 `|` 分隔）时，该表格在块中渲染为视觉格式的表格——而不是原始的管道分隔文本。

2. 表格渲染时具有：
   - 视觉上独特的标题行。
   - 基于分隔行的对齐列：`|:---|` 左对齐，`|:---:|` 居中，`|---:|` 右对齐。没有冒号的 `|---|` 回退到默认对齐（文本左对齐，类似数值的数值右对齐）。
   - 与活动主题一致的可见面行分隔符（或等效间距）。

3. 单元格内的内联 Markdown 以内联方式渲染：粗体、斜体、内联代码、删除线和链接的渲染方式与周围块输出中的相同。单元格内的换行符（`<br>` 或转义的 `\\n`）渲染为单元格内换行。

4. 列宽选择以适应表格的自然内容，当内容适合块内时。如果单个单元格的内容非常长，该单元格会在其列内换行文本，而不是强制列到不合理的宽度。
   - **开放问题：** 当换行的单元格会产生不合理高度的行时，我们是使用"展开" affordance 进行裁剪，还是让行无界增长？

5. 水平滚动：当表格的总宽度超过块宽度——许多列，或无法合理收窄的宽列——表格在块内变为水平可滚动。水平滚动显示屏幕外列，而不剪切或截断它们。块的垂直滚动独立于表格滚动工作。

6. 当块调整大小时（终端调整大小、窗格拆分、侧边栏打开/关闭），表格重新流式传输到新宽度，而不丢失行或列的顺序。

7. 空单元格渲染为可见为空（与周围单元格相同的行高，无占位符文本）。所有单元格都为空的行仍渲染为一行。

8. 只有标题和分隔符（零个正文行）的表格渲染为仅标题表格，而不是原始文本。

9. 单列表格渲染为单列表格（不折叠为项目符号列表或类似内容）。

10. 格式错误的表格优雅回退：
    - 缺少分隔行 → 渲染为预格式化文本，而不是表格。
    - 参差不齐的行（有些行的单元格比标题少或多）→ 缺失单元格渲染为空；显示额外单元格，如果可能则视觉扩展标题行。块绝不应悄悄丢弃数据。
    - 未闭合的表格（最后一行在流中途被截断）→ 渲染为部分表格；见 (11)。

11. 流式输出：当命令仍在产生行时，表格增量渲染。新行在到达时追加。标题行在接收到分隔行时锁定；分隔行之前的行渲染为纯文本，直到表格被识别。

12. 选择和复制：
    - 使用鼠标或键盘跨单元格选择会选择其可见文本内容。
    - 复制选择默认产生制表符分隔的纯文本（每行一行，单元格用制表符分隔）。一个 affordance（上下文菜单、快捷键）让用户改为复制原始 Markdown 源代码。
    - 复制整个块保留原始 Markdown 源代码逐字。

13. 块内搜索（在块中查找）匹配单元格文本内容。匹配在渲染的单元格中就地高亮；导航匹配水平滚动表格到视图，包括匹配在屏幕外列中时。

14. 共享或导出块（云存储、共享链接、保存为文件）保留原始 Markdown 源代码，而非渲染形式。

15. 主题设置：表格边框、标题背景、交替行着色（如果有）以及链接/代码样式都来自活动主题。没有硬编码的颜色。

16. Markdown 表格在块列表 Markdown 已渲染的任何地方一致渲染——命令输出、代理响应以及支持内联 Markdown 的任何其他块类型。相同输入在每个表面产生相同的表格。

17. 非表格管道内容不会被错误渲染为表格。包含 `|` 字符但没有有效标题-分隔行的文本保持为纯文本，即使它视觉上类似于表格。
````
